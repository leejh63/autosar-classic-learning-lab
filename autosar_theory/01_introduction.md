# 01. AUTOSAR 개요 — 무엇을, 왜

## 한 줄 정의

AUTOSAR(AUTomotive Open System ARchitecture)는 차량용 전자제어장치(ECU)의 소프트웨어를
**표준화된 계층 구조와 인터페이스, 그리고 설정 기반 개발 방법론**으로 규정한 개방형
표준이다. 목표는 단순하다: 자동차 SW 를 **하드웨어와 벤더로부터 독립적으로, 재사용
가능하게, 교환 가능하게** 만드는 것.

## 왜 생겼나 — 해결하려는 문제

2000년대 초, 차량의 기능이 폭발적으로 늘면서 ECU 소프트웨어가 다음 문제에 부딪혔다:

- **하드웨어 종속**: 같은 기능(예: 도어 제어)을 MCU 가 바뀔 때마다 다시 짰다. 코드가
  레지스터에 박혀 있어 이식이 거의 새 개발이었다.
- **재사용 불가**: OEM·공급사·차종마다 SW 가 일회성이라 같은 로직을 수없이 중복 개발.
- **통합 지옥**: 여러 공급사가 만든 ECU/모듈을 한 차량 네트워크에 통합할 때 인터페이스가
  제각각이라 결합 비용이 폭증.
- **확장성 한계**: 기능 추가가 곧 전체 재검증. 분업·병렬 개발이 어려웠다.

AUTOSAR 의 답은 "**경쟁은 구현에서, 협력은 표준에서**"다. 인터페이스와 구조를 표준화해
SW 를 모듈로 끊고, 그 모듈을 하드웨어/벤더 너머로 재사용·교환할 수 있게 한 것.

## 컨소시엄

2003년 BMW, Bosch, Continental, Daimler, Volkswagen, Siemens VDO 등이 설립한 글로벌
파트너십. OEM(완성차), Tier-1(공급사), 반도체/툴 벤더가 함께 표준을 만든다. 핵심
산출물은 코드가 아니라 **명세서(specification) 문서**다. 각 회원사가 그 명세에 맞춰
자기 구현(MCAL, BSW 스택, 툴체인)을 만들어 판다.

> 그래서 "AUTOSAR 를 쓴다"는 건 보통 "어떤 벤더의 AUTOSAR 호환 스택 + 설정 툴을
> 쓴다"는 뜻이다. 표준 자체는 무료로 공개된 PDF 명세이고, 실제 동작 코드는 벤더
> 제품이다.

## 세 개의 플랫폼

AUTOSAR 는 용도가 다른 세 갈래로 나뉜다.

**Classic Platform (CP)**
전통적 ECU(엔진·바디·섀시·도어 등) 용. 실시간·결정론·낮은 자원이 핵심. C 언어,
정적 구성, OSEK 계열 RTOS 위에서 돈다. 마이크로컨트롤러(MCU) 직결. 이 핸드북의
주 대상이며, 양산 ECU 펌웨어의 대다수가 여기 속한다.

**Adaptive Platform (AP)**
ADAS/자율주행·인포테인먼트·고성능 컴퓨팅 용. POSIX 운영체제(예: 리눅스/QNX) 위에서
C++ 로 돌고, **서비스 지향(SOA)** 통신을 한다. 동적 배포·업데이트가 가능. 고성능
프로세서(MPU/SoC) 대상. → 자세히는 `12_classic_vs_adaptive.md`.

**Foundation**
CP 와 AP 가 **공유하는 공통 요소**(예: 공통 통신 프로토콜 명세 PRS, 메타모델 공통부,
방법론 공통부). 두 플랫폼이 함께 쓰는 토대라 별도 플랫폼으로 분리되어 있다.

R19-11 이후 세 플랫폼은 매년 11월 **하나의 릴리스로 동시 발행**되며 같은 스키마
버전을 공유한다(→ README 릴리스 표).

## 표준화의 두 축: 구조 + 방법론

AUTOSAR 를 처음 배울 때 흔히 "계층 구조"만 떠올리지만, 실제로 표준의 절반은
**방법론(methodology)** 이다.

1. **구조(architecture)**: 소프트웨어를 어떻게 층으로 나누고 모듈로 끊을지. → `02`
2. **방법론(methodology)**: 그 구조를 **설정(ARXML)으로 기술하고 코드를 생성**하는
   개발 과정. SWC 와 ECU 를 모델로 기술하면 툴이 RTE/BSW 설정을 뽑아낸다. → `03`

이 두 번째 축 때문에 AUTOSAR 개발은 "코드를 많이 짜는" 일이라기보다 "**모델을
설정하고 생성하는**" 일에 가깝다. 직접 베어메탈로 펌웨어를 짜던 방식과 가장 크게
다른 지점이 여기다.

## 핵심 개념 미리보기 (이후 장의 예고)

- **VFB (Virtual Function Bus)**: SWC 들이 "가상의 버스"로 연결된 것처럼 설계하고,
  실제 배선(어느 ECU·어느 버스)은 나중에 매핑한다는 추상. → `04`, `05`.
- **SWC (Software Component)**: 재사용 단위의 애플리케이션 소프트웨어. 하드웨어를
  모른다. → `04`.
- **RTE (Runtime Environment)**: VFB 를 특정 ECU 에서 실제로 실현하는 생성된 미들웨어.
  SWC 의 유일한 통로. → `05`.
- **BSW (Basic Software)**: 통신·진단·메모리·OS·서비스 등 표준 인프라 계층. → `06`~`09`.
- **MCAL (Microcontroller Abstraction Layer)**: 레지스터를 만지는 최하위. → `02`, `09`.

## AUTOSAR 가 약속하는 것과 치르는 비용

얻는 것:
- **이식성**: SWC 를 다른 ECU/MCU 로 옮길 때 SWC 코드는 그대로, 설정만 바꾼다.
- **재사용**: 검증된 BSW/SWC 를 차종·프로젝트 너머로 재활용.
- **분업**: 인터페이스가 표준화돼 OEM/공급사/팀이 병렬 개발·통합.
- **교환성**: 같은 표준을 따르면 다른 벤더 구성요소로 교체 가능.

치르는 비용(솔직히):
- **설정 복잡도 폭발**: 모듈마다 수많은 파라미터. 툴 없이는 사실상 불가능.
- **툴 종속**: ARXML·generator·벤더 IDE 에 강하게 의존.
- **추상화 오버헤드**: 층이 깊어 코드·RAM·지연 비용이 베어메탈보다 크다.
- **학습 곡선**: 모듈 수백 개 + 방법론 + 메타모델. 진입장벽이 높다.

이 trade-off 를 의식하는 게 중요하다. AUTOSAR 는 "대규모 분업·장수명·다차종" 맥락에서
비용 대비 이득이 크지만, 소형 단일 프로젝트엔 과할 수 있다. 표준이 왜 이렇게까지
복잡한지를 이 비용/이득 관점에서 보면 납득이 쉬워진다.
